# ADR-0036: Playwright TypeScript as the E2E Acceptance Testing Layer for Portal Features

**Status**: Accepted  
**Date**: 2026-04-08  
**Deciders**: Architecture Team

## Context

### The existing test pyramid and its gap

OSM2's portals (`osm2-taxable-person-portal` on port 8090, `osm2-authority-portal` on
port 8091) are Thymeleaf + HTMX BFFs. The existing test layers are:

| Layer | Technology | Scope | Infrastructure |
|-------|-----------|-------|----------------|
| Unit tests | JUnit 5 + Mockito | Individual classes, controller method return values, template assertions | None — in-process |
| BDD acceptance tests | Cucumber-JVM 7.x | Spring-layer behaviour, service integration, business rules | H2 or embedded PostgreSQL — in-process Spring context |
| Integration tests | JUnit 5 + maven-failsafe-plugin | Flyway migrations, immudb connectivity, Keycloak integration | Docker containers for specific dependencies |

None of these layers exercise the portals **through a browser**. Portal features can pass
all Cucumber scenarios and still fail in production due to:

- Thymeleaf fragment rendering errors masked by browser error recovery
- HTMX `hx-*` attribute mismatches (wrong target IDs, swap strategies)
- Keycloak OIDC redirect flows behaving differently with a real browser vs a mocked security context
- WCAG violations invisible to JVM-level tests
- JavaScript errors silently blocking form submission

### Prior Playwright exploration

The `.playwright-cli/` directory at the repository root contains a page capture from
2026-03-30, confirming that Playwright was already being used for ad-hoc exploratory
testing. This ADR formalises that practice into a systematic, pipeline-integrated layer.

### Toolchain context

`node = "22"` is already declared in `.mise.toml` with the comment
*"playwright-cli and other JS dev tooling"*. Node.js is available to all developers and
in CI (via `jdx/mise-action`) without any additional toolchain change.

### Technology options evaluated

| Option | Assessment |
|--------|-----------|
| **`playwright-java` Maven module** | Stays in the Java ecosystem, but Maven lifecycle (surefire/failsafe) is the wrong layer for browser tests against a live stack. `spring-boot:start` Maven plugin is fragile for starting two portals + Keycloak + PostgreSQL. Adds Maven module overhead (Spotless, JaCoCo, parent POM) for a test-only concern. playwright-java has thinner tooling than TypeScript. |
| **Standalone `osm2-e2e/` Node.js TypeScript project** | Playwright's primary and most documented language. `docker compose up` + `npx playwright test` mirrors production deployment. Node.js is already in the toolchain via mise. The `playwright-test-generator` agent generates tests from Gherkin feature files. |
| **Selenium WebDriver (Java)** | Inferior auto-waiting, no built-in trace viewer. Playwright is the successor and current industry standard. |
| **Cypress** | JavaScript-only, iFrame/multi-tab limitations, multi-domain constraints problematic for Keycloak OIDC redirect flows. Commercial cloud offering. |

## Decision

We adopt **Playwright TypeScript** in a standalone **`osm2-e2e/` Node.js project** as
the E2E acceptance testing layer for all portal features.

### Test pyramid positioning

```
                    ┌─────────────────────────────────────┐
                    │  E2E (Playwright TypeScript)         │  osm2-e2e/
                    │  Browser → Portal → Services         │  docker compose stack
                    └─────────────────────────────────────┘
              ┌─────────────────────────────────────────────────┐
              │  BDD Acceptance (Cucumber-JVM)                  │  in-process Spring
              │  Spring context → H2 / embedded PG              │  per Maven module
              └─────────────────────────────────────────────────┘
    ┌──────────────────────────────────────────────────────────────────┐
    │  Unit (JUnit 5 + Mockito) + Integration (maven-failsafe)         │
    └──────────────────────────────────────────────────────────────────┘
```

### Project structure

`osm2-e2e/` is a sibling directory to the Maven modules and is **not** listed in
`pom.xml`. Maven ignores it entirely.

```
osm2-e2e/
├── package.json               # @playwright/test, typescript
├── playwright.config.ts       # baseURL per portal, projects (chromium)
├── tsconfig.json
└── tests/
    ├── taxable-person-portal/
    │   └── *.spec.ts
    └── authority-portal/
        └── *.spec.ts
```

Test files are generated by the `playwright-test-generator` agent from Gherkin feature
files. The agent reads `.factory/project.yaml` for the `e2e` section to locate
`project_root`, `test_dir`, and `base_urls`.

### CI integration

The `e2e-tests` CI job depends on `build`. It uses `jdx/mise-action` (already present
in all other jobs) to supply Node.js 22 from `.mise.toml` — no additional toolchain
setup is required.

The job:
1. Builds JARs (`mvn --batch-mode package -DskipTests`)
2. Starts the full stack (`docker compose up -d`)
3. Waits for `/actuator/health` on both portals
4. Installs Playwright chromium (`npm ci && npx playwright install --with-deps chromium`)
5. Runs `npx playwright test`
6. Uploads the HTML report to GitHub Artifacts (30-day retention)
7. Runs `docker compose down -v` in `always()` to clean volumes

Chromium only in CI. Firefox and WebKit are opt-in for local developer runs.

### Test generation

Portal Gherkin scenarios reaching Phase 5 of the delivery pipeline are routed to
`playwright-test-generator` (not `bdd-test-generator`) when `playwright.config.ts` is
detected via `.factory/project.yaml`. Tests are generated in RED phase —
`throw new Error('Not implemented: <petition-ID> — "<scenario title>"')` — and turned
green when a developer implements the Playwright actions.

## Consequences

### Improved

- Portal features are verified end-to-end: rendering, HTMX behaviour, Keycloak OIDC
  redirect flows, and cross-cutting accessibility concerns.
- Formalises the Playwright exploratory work already present in `.playwright-cli/` into
  a systematic, pipeline-integrated test layer.
- Node.js is already in the toolchain via mise — no new developer toolchain requirement.
- The `playwright-test-generator` agent generates traceable `test()` blocks from Gherkin
  scenarios, consistent with the petition pipeline's `validation-contract.md` artefacts.
- Playwright trace viewer and screenshot capture provide evidence for compliance audit.

### Accepted costs

- **E2E tests require a running stack.** They cannot run in the pre-commit or unit-test
  phase. CI time increases due to `docker compose up` and portal health checks.
- **Two test authoring languages.** Cucumber steps are Java; Playwright tests are
  TypeScript. The TypeScript subset required (test(), expect(), page object) is minimal.
- **Test brittleness risk.** Browser tests are more brittle than unit tests. Playwright's
  auto-waiting and strict selectors mitigate this, but E2E suite maintenance is an
  ongoing commitment.

### Unchanged

- `pom.xml` (parent and modules) has no knowledge of `osm2-e2e/`.
- Cucumber-JVM remains the primary BDD layer for business rule acceptance.
- ADR-0032 (Catala oracle) is unaffected.
- ADR-0023 (SKAT design system) is unaffected; Playwright tests against rendered HTML
  regardless of the server-side templating technology.
